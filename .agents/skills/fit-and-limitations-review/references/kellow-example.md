# Kellow Example

A fictional scenario built to test the fit-and-limitations-review skill on a prospect where more than one team is in scope, and they do not all fit equally well.

## Task

A discovery call at Kellow Distribution Group covered three teams: Regional Account Managers (named individual accounts, one-to-one calls, existing CRM record), Central Order Support (a shared queue, no call ownership, a separate order management system with unconfirmed integration), and Field Service Engineers (mentioned in passing, with no actual workflow detail established). Classify each.

## Deliberate Traps

- Regional Account Managers is a clean good fit and should be said so plainly, without inventing caveats to look more balanced.
- Central Order Support's shared-queue structure is temptingly easy to reframe as an advantage ("one rollout reaches the whole team"), when the real story is an unresolved ownership and integration mismatch. State the mismatch, not the spin.
- Field Service Engineers has no real evidence either way. "They probably have the same problem" is a hypothesis, not proof of fit, and the honest classification is uncertain, not a soft poor fit.

## Abbreviated Good Output

- **Regional Account Managers**: good fit, confirmed integration, named contact, one call to one CRM record.
- **Central Order Support**: poor fit, shared ownerless queue plus an unconfirmed separate system, stated as a real mismatch, not an adoption advantage.
- **Field Service Engineers**: uncertain, no workflow evidence established, next step is finding out rather than guessing.

## Conclusions the Skill Must Not Make

- That Central Order Support's shared structure is a hidden advantage rather than a real integration and ownership problem.
- That Field Service Engineers is a good fit, or a poor fit, when neither has actually been established.
- That any capability not explicitly confirmed (the order management system integration) can be assumed to work because it seems plausible.

Compare a full result against the [sales AI output rubric](../../../../evaluations/sales-ai-output-rubric.md), checking in particular that the poor-fit classification never reads as a disguised strength.
